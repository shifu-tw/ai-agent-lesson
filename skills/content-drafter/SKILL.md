# Content Drafter

## 觸發條件

使用者說以下任一：
- 「幫我寫 ___」
- 「起草 ___」
- 「draft ___」
- 「幫我回覆 ___」
- 「幫我回這封信」

## 執行步驟

### Step 1：確認需求

如果使用者沒說清楚，快速確認：
- **寫什麼**：email / 訊息 / 貼文 / 文件 / 其他
- **給誰看**：對象是誰（決定正式程度）
- **目的**：通知 / 說服 / 回覆 / 請求 / 其他
- **長度偏好**：簡短 / 中等 / 詳細

如果使用者已經說得很清楚，不要再問。直接寫。

### Step 2：讀取風格設定

從 CLAUDE.md 讀取：
- 溝通偏好（條列/段落、語氣、用詞習慣）
- 思考框架（決策邏輯、在意的面向）
- 紅線（不要做的事）

從 MEMORY.md 讀取：
- 過去對內容風格的糾正紀錄
- 對特定人/場景的溝通偏好

### Step 3：起草

根據以上資訊起草。規則：
- **用使用者的口吻**，不是 AI 的通用語氣
- **結論先行**：最重要的事放第一段
- **簡潔**：能一句講完不要兩句
- **不加 AI 味的客氣話**：不要「希望您一切順利」「感謝您的來信」這類模板開頭
- **保留使用者的思考方式**：如果他習慣先給結論再解釋，就這樣寫

### Step 4：產出

直接給出可以用的稿件。不要加「以下是草稿：」之類的前綴。

如果有多種可能的方向，給一個主版本 + 簡短說明為什麼這樣寫。
不要給三個版本讓使用者選 — 給你判斷最好的那一個。

### Step 5：等待回饋

使用者可能說：
- 「太正式了」→ 調整語氣，記進 MEMORY.md
- 「太長了」→ 精簡，記住長度偏好
- 「OK」或直接用 → 這個風格是對的，記住
- 「改一下 ___」→ 修改，判斷是否值得記住

## 品質標準

- 讀起來像使用者自己寫的，不是 AI 寫的
- 如果使用者一次就用了不太需要改 → 成功
- 如果每次都要大改 → 需要更新 CLAUDE.md 的溝通偏好
